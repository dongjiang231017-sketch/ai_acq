## 素材要点（已通读，直接用这些结论）

- 五条铁律：开场 15 秒说清三件事；一句话不超一口气(约20字)说完就停；客户开口立刻闭嘴；**价格不主动报**，问到才报且立刻回到价值；所有分支收口到"您这个号就是微信吧？我加您，把案例发您看"。
- 增补篇原则：**不报服务价**；抽成话术只讲**千分之六**；不冒充腾讯官方；不承诺单量/效果；不贬低美团抖音只讲增量。
- 增补篇有 7 条带 ⚠（公司政策/AI 人设占位符）**未经确认不得入库**——zip 里缺的编号就是它们（如 56、57、58、60 等，以实际解压为准），执行时列出缺号清单放进验收报告。

## 任务

### 1. 话术结构化入库
解析两个 docx，抽取为结构化条目：编号、类别（开场/主流程/问答A-G/收尾/特殊）、客户触发语、话术原文、执行提示（▸ 开头的行）。导入平台话术库（`/api/outbound/scripts` 即 CallScript，或按现有话术管理的正确姿势），保留 编号→音频文件名 的映射字段。**不要删除旧话术**，旧话术标记停用即可，保留回滚能力。

### 2. 实时通话话术切换（LiveKit omni 路线）
重写 `backend/app/services/realtime_sales_playbook.py` 的 `build_video_group_buying_sales_instructions`：以新手册为唯一依据（五条铁律 + 开场A为默认开场 + 主流程1-5 摸底/算账/三板斧/全托管/收尾 + 全部异议分支应对 + 收口规则），并硬性写入：不报服务价、抽成只说千分之六、不冒充官方、被问是否 AI 才承认。注意 instructions 总长度对首包延迟的影响，控制在合理范围（可分层：核心规则全文 + 异议库摘要）。

### 3. 克隆语音接入
- 解压 zip 到平台语音素材目录（`.voice_samples` 体系），按现有语音包/音频缓存机制（`realtime_voice_cache_*` 配置、语音包管理）导入 74 条音频，建立 话术条目→音频 映射，供 pipeline 路线和兜底播报直接放音。
- 确认这批音频对应的克隆音色在平台里是否已有 voice id（voice_clone 体系）。如有：把默认外呼音色切到该克隆音色。
- **限制说明**：LiveKit omni 实时路线的声音由 DashScope omni realtime 内置音色决定（当前 Aiden），如 DashScope 不支持在 omni realtime 会话里使用克隆音色，则：omni 路线保持内置音色 + 开场白等固定话术优先用克隆音频放音；把"整通克隆音色"能力落在 pipeline 路线。把实际能做到的程度写进验收报告，不要硬造不存在的 API。

### 4. 验收（拨 18850625208）
- 开场必须是新手册开场A原文风格："喂，老板您好！耽误您半分钟。我是做视频号团购的……咱店里这块开了没？"
- 问"多少钱"→ 不报价，回到价值并收口加微信；问"抽成"→ 只说千分之六。
- 声音：能用克隆音色的环节用克隆音色，并在报告里注明哪些环节是克隆音色、哪些是内置音色。
- 验收报告写进 `docs/plans/2026-07-10-script-voice-import-report.md`。
